本記事では、アプリケーションの「インストール」とは何か、その役割と様々な方法について解説します。

## インストールとは何か

インストールとは、コンピュータにアプリケーションを導入して「**使える状態にすること**」です。

多くの場合、単にファイルをコピーするだけではアプリは動きません。  
設定ファイルの配置や、OSへの登録など、環境を整えるための処理が必要になります。

つまり、アプリの導入には、大きく分けて **「ファイルのダウンロード」と「使えるようにするための設定」の2つの工程が必要です**。  
このうち、後者の **「使えるようにするための設定」をインストール**といいます。

ただし、アプリストアなどでは「ダウンロード」と「インストール」をまとめて一つの処理として扱うことも多く、文脈によって少し意味が変わる場合があります。

### インストールの役割 - 使える状態とは

上でも書いたように、基本的にアプリはファイルを置いただけでは動きません。  
実行に必要なライブラリや設定などを整え、OSから認識されるようにする必要があります。

インストールの役割は、この「動作に必要な環境を整えること」です。  
その処理は、見えないところでいくつかの段階に分かれて実行されています。

1. **ファイルの配置**  
    実行ファイルや設定ファイルを、OSが参照できる場所に配置します。
2. **依存関係の準備**  
    動作に必要なライブラリやモジュールを導入します。（`DLL`や`.so`など）
3. **システムへの登録**  
    ショートカットやアンインストール情報の登録、レジストリ設定などを行い、OSがアプリを認識できるようにします。
4. **環境設定・初期化**  
    ユーザ設定フォルダを作成したり、初期データを展開します。

このようにインストールは、アプリを「動作環境に合わせて、動かせる状態に整える」ためのステップです。

## インストールの方法

「インストール」といっても、そのやり方はアプリによって様々です。  
しかし、インストールの方法は大きく次の4つに分類することができます。

1. **GUIインストーラー**
2. **アプリストア**
3. **コマンド**
4. **ファイルコピー**

それぞれメリットがあり、使い分けが大事です。

### 1. インストーラー方式（`.exe` / `.msi` / `.pkg`など）

特にWindowsでよく使われる、ファイルのダブルクリックなどで実行する方式です。

管理者権限でインストールする場合も、ユーザは特に意識せずに**UACプロンプト**などを用いて権限を昇格させることができます。  
UACプロンプトとは、次の画像のようなWindowsのユーザアカウント制御（User Account Contorol）機能です。

インストーラーで主に使われるファイルは、Windowsの`.exe`や`.msi`、macOSの`.pkg`ファイルです。

`.exe`はインストーラーの実行ファイルで、インストーラー本体です。

対して`.msi`や`.pkg`は実行ファイルではなく、**OSが持っているインストーラーに参照されるファイル**です。  
ファイルはインストールに必要なデータのみを保持し、インストーラー本体はOSが持っているものを使います。

インストーラーは、上で説明した「ファイル配置 -> 依存関係 -> 登録 -> 初期化」といった処理を**まとめて行ってくれる**便利な仕組みです。

また、インストールの失敗や中断時に元の状態に戻してくれる機能なども備えており、コンピュータの知識がない人でも安心してインストールできる方式です。

### 2. アプリストア（AppStore / GooglePlayなど）

スマホでお馴染みの方式で、最近はPCでも普及しています。

アプリストアが**ダウンロード・インストール・アップデートをまとめて面倒見てくれる**ため、ユーザは操作をほとんど意識しなくて済みます。

この方法もインストーラー方式と同様に、初心者が安心してインストールできる方式になっています。

特に、ストアがセキュリティ的なチェックをしている点や、全てのアプリ管理がストア経由で完結する点が特徴です。

### 3. コマンド（`apt` / `brew` / `npm` / `pip`など）

主に開発者向けのアプリやライブラリで多く使われる方式です。

ターミナルでコマンドを実行すると、必要なファイルのダウンロードから依存関係の解決、環境への登録までを自動で行ってくれます。

再現性の高い環境構築ができ、**自動化しやすい**という点があります。

インストールに必要なコマンドをまとめて`.sh`などのシェルスクリプトとして実行できるようにする方法や、ソースコードを自分でダウンロードして`configure`や`make`などで自分でビルドとインストールを行う方法もあります。  
このシェルスクリプトや`configure`、`make`については、後日、別の記事でまとめようと思っています。

### 4. ファイルコピー（`.zip`や`.dmg` / ポータブルアプリ）

最もシンプルな方式で、ZIPなどにアプリ本体が入っており、そのアプリをフォルダに置くだけの方式です。

私の体感ですが、macOSではこの方式が多く、`.dmg`のなかに`.app`というアプリ本体が入っていることが多いです。

macOSの`.app`ファイルは、実行ファイルやリソースを全て含んだ**バンドル**と呼ばれる単一のパッケージ構造になっています。  
これにより、特別な設定なしにファイルをコピーするだけで実行が可能です。

また、**ポータブルアプリ**もこれに該当します。  
ポータブルアプリとは、PCにインストールすることなく、USBメモリなどの外部記憶装置に入れて持ち運び、別のコンピュータでそのまま実行できるアプリのことです。

この方法は、導入が早くアンインストールも簡単という特徴があります。

## インストール方法の本質的な違い

大きく4つの分類でまとめてみましたが、なぜ分かれているのでしょうか。

それは、**アプリがユーザーに要求できる技術的な知識のレベルがそれぞれ異なるから**です。

インストール方式の違いを一言でまとめると「**どこまで自動化し、どこまでをユーザに任せるか**」です。

例えば、大規模なゲームと小さいライブラリではユーザにとって求めるものが違います。  
大規模なアプリはインストーラーなどで安全にインストールしたいと思う反面、小規模なライブラリは手軽にコマンドで解決できた方が便利でしょう。

また、ユーザの違いもあります。  
コンピュータに慣れていないユーザは、ダブルクリックやタップだけで進められる方法がいいでしょうし、プログラマやシステム管理者は自動化のしやすさやカスタマイズのしやすさを求めます。

他にも、OSの違いもあります。  
WindowsはレジストリやProgramFilesという文化を持っていますが、macOSはアプリバンドルという文化を持っており、アプリの存在の仕方が異なります。

このように、開発者視点では**ターゲットユーザやアプリの特性を考慮し、最適なインストール方法を選択することが重要**です。
