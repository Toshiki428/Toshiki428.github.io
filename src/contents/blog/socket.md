本記事では、TCP通信を前提として、「**ソケット**」についてまとめています。

## 事前知識

### 1. FD（ファイルディスクリプタ）

ユーザプロセスがシステムコールをするときに、**対象となるファイルを指定する数字**です。

簡易的な説明になりますが、Linuxにはディスプレイやキーボードなども全てファイルとして扱う「**Everything is a file**」という考え方があります。  
もちろん、ネットワークもファイルとして扱うことになります。

つまりLinuxは、FDを使い「接続相手」を管理しています。

FDについて、詳しくは[こちらの記事](/#/blog/file-descriptor)で解説しています。

### 2. ssコマンド

`ss`とは、`Socket Statistics`の略で、直訳すると「ソケットの統計」になります。

**ソケットの状態**や**ソケットを使っているプロセス**などを知りたい時に使われます。

昔は`netstat`というコマンドが主流でしたが、現在はより高速で詳細な情報が出せる`ss`コマンドが主流になっています。

## ソケットとは

コンピュータの通信では「IPアドレス」と「ポート番号」を使い、通信相手を特定します。  
OSは、これらの情報などを含んだ「**通信の状態**」を内部で管理しており、その管理データを**ソケット**と呼びます。

ソケットを使うことで、アプリは通信相手や接続状態を意識しすぎることなく、通信を扱えるようになります。

### ソケットの中身

例えば、OSが次のようなテーブルを持っていると考えてみてください。

| ID  | 自分のIP      | 自分のポート | 相手のIP        | 相手のポート |
| --- | ------------- | ------------ | --------------- | ------------ |
| 3   | 192.168.1.100 | 8080         | 192.168.1.**1** | **54321**    |
| 4   | 192.168.1.100 | 8080         | 192.168.1.1     | **65432**    |
| 5   | 192.168.1.100 | 8080         | 192.168.1.**2** | 54321        |

自分のIPとポートが同じでも、相手のIPやポートが違えば、OSはこれらを**別の通信**として区別できます。  
そして、アプリは「自分のIPとポート、相手のIPとポート」を管理せずに、**IDを指定すれば特定の相手と通信**できます。

このIDが「**FD（ファイルディスクリプタ）**」と呼ばれるものです。

事前知識のセクションで触れたように、Linuxではネットワーク通信もファイルとして扱われます。  
OSは、内部で管理する各ソケットに**FD**という一意の番号を割り当てることでこの思想を実現しており、そのおかげで、アプリケーションはファイルに書き込んだり読み込んだりするのと同じ**統一された方法で、ネットワークデータの送受信ができる**のです。

そして、このFDには専用の **「送信用」と「受信用」のバッファ** がOSによって紐づけられています。  
アプリケーションがデータを受信する際は、OSが相手の情報を基に適切なFDの受信バッファにデータを振り分けます。  
そのおかげで、アプリケーションはその**FDを指定してバッファからデータを読み出すだけ**で、特定の相手からのデータを簡単に受け取れるというわけです。

ソケットには大きく、「待ち受け（LISTEN状態）」と「通信（ESTABLISHED状態）」があります。  
LISTENのソケットは全体からのアクセスを受け付け、ESTABLISHEDのソケットは個別の通信を担当します。

個別の通信中でも他の新規接続に対応できるよう、このように役割が分けられています。

## 実演

ソケットの理解を深めるために、PythonとLinuxコマンドを使って「OSが管理しているソケットやFDが接続の前後でどう変化するのか」を実演しながら確認してみます。

> 本文ではLinux環境での確認方法を示します。  
> Windows / macOSの場合は、対応するコマンドに読み替えてください。

### 実演の準備

ここでは、後の実演に向けて「実演に使うプログラム」と「最低限知っておくべきポイント」だけを確認します。  
詳細な挙動や仕組みの解説は、実演後に改めて行います。

**今回使うサーバコード**

```python
import socket
import os

server_socket = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server_socket.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
server_socket.bind(('0.0.0.0', 8080))
server_socket.listen(5)

print(f"【LISTEN起動】PID: {os.getpid()} / FD: {server_socket.fileno()}")
print("別ターミナルで 'ss -lnt' を打って、8080番がLISTENなのを確認します。")
input(">>> 次へ進むには Enter (接続待ち状態へ)")

try:
    while True:
        print("\n--- 接続を待っています（accept中） ---")
        conn, addr = server_socket.accept()
        
        new_fd = conn.fileno()
        print(f"【ESTAB起動】相手: {addr} / FD: {new_fd}")
        
        print(f"\n--- いま接続が確立されました ---")
        print(f"別ターミナルで 'ss -nt' や 'ls -l /proc/{os.getpid()}/fd' を打って！")
        print(f"FD {new_fd} が増えているはずです。")
        
        input(">>> メッセージを受信するには Enter")
        
        data = conn.recv(1024)
        if data:
            print(f"受信データ: {data.decode().strip()}")
        
        print(f"\nもう一度 `ss -nt` で確認するとRecv-Qが0になっているはずです。")
        input(">>> 接続を閉じるには Enter")
        conn.close()
        print(f"【子終了】FD {new_fd} を閉じました。")

finally:
    server_socket.close()
```

**実演前に押さえておくべきポイント**

1. `server_socket`  
    LISTEN通信用のソケットです。

2. `server_socket.bind(('0.0.0.0', 8080))`  
    ここでは、通信を受け付けるネットワークインタフェース（通信を受け取る入口）を指定します。  
    コンピュータは複数のネットワークインタフェースを持っています。  
    無線LAN子機（Wi-Fiアダプタ）、LANポートそれぞれにNICを持っており、localhost用のインタフェースもあります。  
    `'0.0.0.0', 8080`とは、「どのインタフェースから入ってくる通信でも、`8080`番のポート宛の通信なら受け取る」という設定です。  
    `0.0.0.0`の部分を`127.0.0.1`と書けば、localhostからの通信のみを受け付けるようになります。

3. `server_socket.listen(5)`  
    これは、「受け入れ可能な接続待ち行列の最大数」を設定しています。  
    今回の場合、「5人までは並ばせていい、6人目以降は断る」という設定です。  
    「ソケットの最大接続数」ではありません。  
    クライアントとの通信は`.accept()`によって開始されますが、`accept()`されていない接続が、最大でいくつ並べるかを指定しています。

### 実行

では、実行してみます。  
はじめに、このように表示されました。

```txt
【LISTEN起動】PID: 27 / FD: 3
別ターミナルで 'ss -lnt' を打って、8080番がLISTENなのを確認します。
>>> 次へ進むには Enter (接続待ち状態へ)
```

では、ここで別ターミナルを開き、書かれているコマンドを実行してみます。

**1. LISTENソケットの確認**

```sh
# -l: State=LISTENのみを表示
# -n: 名前解決せず数字で表示
# -t: TCPソケットを表示
ss -lnt
```

| State  | Recv-Q | Send-Q | Local Address:Port | Peer Address:Port |
| ------ | ------ | ------ | ------------------ | ----------------- |
| LISTEN | 0      | 5      | 0.0.0.0:8080       | 0.0.0.0:*         |

`Peer Address:Port`の`0.0.0.0:*`は、**あらゆる接続元からの通信を受け付ける**ワイルドカードを示します。  
環境によっては`*.*`や`::*`のように表示される場合もあります。

LISTENソケットとESTABソケットでは、`Recv-Q`と`Send-Q`の意味が異なります。  
詳しい意味の違いは、後の解説セクションで説明します。

**2. クライアントからの接続**

では、Python実行中のターミナルに戻って、`Enter`を押します。  
すると、`--- 接続を待っています（accept中） ---`となるので、  
別ターミナルから次のようなコマンドで接続してみましょう。

```sh
curl localhost:8080
# または
nc localhost 8080
```

Python実行中のターミナルに戻って、次のように表示されていれば接続成功です。

```txt
【ESTAB起動】相手: ('127.0.0.1', 34522) / FD: 4

--- いま接続が確立されました ---
別ターミナルで 'ss -nt' や 'ls -l /proc/27/fd' を打って！
FD 4 が増えているはずです。
>>> メッセージを受信するには Enter
```

**3. ESTABソケットの確認**

では、ターミナルに書かれている通り、別ターミナルからコマンドを実行します。

```sh
ss -nt
```

| State | Recv-Q | Send-Q | Local Address:Port | Peer Address:Port |
| ----- | ------ | ------ | ------------------ | ----------------- |
| ESTAB | 0      | 0      | 127.0.0.1:8080     | 127.0.0.1:34522   |
| ESTAB | 0      | 0      | 127.0.0.1:34522    | 127.0.0.1:8080    |

上記のように、同じ接続に対して2つのESTABLISHED状態のソケットが表示されます。  
これは、localhost上で通信しているため、サーバー側とクライアント側の両方のソケットが見えているためです。

- **1行目 (`Local Address:Port`が`...:8080`)**  
    Pythonで実行している**サーバー側**のソケットです。`8080`番ポートで待ち受けており、`34522`番ポートのクライアントと通信しています。
- **2行目 (`Local Address:Port`が`...:34522`)**  
    `curl`や`nc`を実行した**クライアント側**のソケットです。OSが自動的に割り当てた`34522`番ポートから、`8080`番ポートのサーバーへ接続しています。

もし、別のコンピュータ（仮想マシンなども含む）から接続している場合は、`ss`コマンドを実行したマシン側のソケットのみが表示されます。

ちなみに`ss`コマンドに`-a`オプションをつけると、LISTENとESTABの両方をまとめて表示できます。

**4. FDの確認**

```sh
ls -l /proc/27/fd
```

```txt
total 0
lrwx------ 1 root root 64  1月 24 05:15 0 -> /dev/pts/0
lrwx------ 1 root root 64  1月 24 05:15 1 -> /dev/pts/0
lrwx------ 1 root root 64  1月 24 05:15 2 -> /dev/pts/0
lrwx------ 1 root root 64  1月 24 05:15 3 -> 'socket:[11268]'
lrwx------ 1 root root 64  1月 24 05:15 4 -> 'socket:[12000]'
```

この結果から、ソケット用に**2つのFDが使われている**ことがわかります。  
これらが「待ち受け用ソケット(FD:3)」と「通信用ソケット(FD:4)」です。なぜ2種類あるのかは、続く「解説」セクションで詳しく説明します。

今回はプログラムが対応していませんが、クライアントからの接続が増えると、通信用ソケットのFD(5, 6, ...)も増えていきます。

### 解説

#### 1. `Recv-Q`と`Send-Q`の意味

LISTENとESTABは、そもそも役割が違います。  
その都合で、Recv-QとSend-Qの意味が変わっています。

**LISTENソケット**
- `Recv-Q`：現在の行列に並んでいる「未処理の接続リクエスト数」
- `Send-Q`：そのポートで受け入れ可能な「接続待ち行列の最大数（Pythonプログラムの`listen()`の引数）」

**ESTABソケット**
- `Recv-Q`：アプリがまだ受け取っていないメッセージの**バイト数**（受信バッファ内のデータ量）
- `Send-Q`：相手に送りきれていない、OSが預かっているメッセージの**バイト数**（送信バッファ内のデータ量）

#### 2. バッファの確認

実際に`Recv-Q`の値を確認します。  
`--- 接続を待っています（accept中） ---`までプログラムを進め、別ターミナルから次のコマンドを実行してください。

```
nc localhost 8080
```

そのまま文字を打ち、Enterを押します。  
そして別のターミナルから`ss -nt`コマンドを実行し、`Recv-Q`を確認してみます。

すると、打った文字分のバイト数が表示されているはずです。

```sh
# nc localhost 8080
あ abc
```

```sh
# ss -nt
State       Recv-Q       ...
ESTAB       8            ...
```

この場合、「あ」=4、「スペース」=1、「abc」=3で、8と表示されました。

数字が増えたことを確認できたら、Python実行中のターミナルに戻り、Enterを押してメッセージを受信してみてください。  
受信後、再び`ss -nt`コマンドを実行すると、`Recv-Q`が0に戻っていることがわかります。

これは、通信相手から受け取ったデータがまずバッファに追加されているということ意味しています。  
このバッファに入っているデータサイズが`Recv-Q`です。  
そして、ソケットの受信バッファに入っているデータは、プログラム側が受け取れたタイミングで破棄されるということが確認できました。

#### 3. `Local Address:Port`と`Peer Address:Port`

`Local Address:Port`は、自分側のアドレスとポート  
`Peer Address:Port`は、相手側のアドレスとポートのことです。

LISTENソケットは次のようになっていました。

| Local Address:Port | Peer Address:Port |
| ------------------ | ----------------- |
| 0.0.0.0:8080       | 0.0.0.0:*         |

自分側はポート番号のみの指定で、`bind`で設定した通りになっています。  
相手側はIPアドレスもポート番号も**指定なし**になっているので、このLISTENソケットはどこからの通信でも受け付けます。

一方、ESTABソケットは次のようになっていました。

| Local Address:Port | Peer Address:Port |
| ------------------ | ----------------- |
| 127.0.0.1:8080     | 127.0.0.1:34522   |

通信を受け付ける場所は`127.0.0.1:8080`、通信相手は`127.0.0.1:34522`であることが確認できます。  
つまり、1対1の通信であるということです。

#### 4. FDが2つある理由

「実演」で確認したように、クライアントが1台接続した時点で、サーバープロセスはソケット用に2つのFDを持つことになりました。  
これは、それぞれが異なる役割を担っているためです。

-   **待ち受け用ソケット（FD: 3）**  
    `socket()`で作成され`listen()`で待ち受け状態になったソケットです。  
    クライアントからの最初の接続要求を受け付けるのが専門の「受付係」です。  
    プログラム内の`server_socket`がこれにあたります。

-   **通信用ソケット（FD: 4）**  
    待ち受け用ソケットが接続要求を受け付け、`accept()`が成功したときに**新しく作られる**ソケットです。  
    実際にクライアントとデータをやり取りするのは、こちらの「担当者」の役割です。  
    プログラム内の`conn`がこれにあたります。

このように、サーバーは「**たくさんのクライアントからの接続を待つ受付係**」と、「**特定のクライアントと実際に通信する担当者**」の役割を分けることで、複数の接続を効率的に管理しています。  
新しいクライアントが接続してくるたびに、この通信用ソケット（担当者）が新しく作られていきます。
